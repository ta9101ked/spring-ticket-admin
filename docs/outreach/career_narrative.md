# 経歴の語り方（営業・面談用）

目的: 「1年ブランク」を弱みの説明で終わらせず、**次の案件につながるストーリー**にする。

## 1. 禁止パターン

- 「辞めてから何もしてません」
- 「ブランクがあるので安くします」（自分から安売り）
- 航空券クーポンを「実装しました」と盛る（設計関与まで）
- fukurou / 実クライアントの機密を具体名で出す

## 2. 推奨ストーリー（30秒版）

1. **以前**: 業務システム寄り（Java/Spring の改修・保守、設計への関与）
2. **その後**: いったん現場を離れたが、エンジニアを捨てたわけではない
3. **直近1年**: PHP/Laravel で業務画面の設計〜実装を継続（店舗運営系の管理画面）
4. **いま**: Spring でも同じ「管理画面の型」を説明可能なサンプルとして公開
5. **これから**: 小さな改修を確実に納品し、継続と単価を上げる

## 3. 60秒版（面談用）

「以前は業務システムの改修・保守を中心にやっていました。設計面にも関わった経験があります。

その後、会社のエンジニア職からは離れましたが、その期間も PHP/Laravel で業務画面の設計と実装を続けていました。一覧・検索・登録・状態管理といった、いわゆる管理画面の課題です。

いまはその型を Java/Spring でも再現し、テストと設計書つきで公開しています。ブランクというより、個人で一気通貫の練習をした期間です。

最初の案件は、8〜20時間で完了が切れる小改修から入り、確実に納品して信頼を積みたいです。」

## 4. 質問への返し方

### Q. ブランク中は何を？

A. 「業務画面の個人開発です。PHP/Laravel で設計から実装までやり、最近は Spring でも同じ粒度のサンプルを公開しています。」

### Q. 実務から離れてない？

A. 「企業常駐からは離れましたが、実装と設計の手は止めていません。公開リポジトリと設計書で確認できます。」

### Q. Java と PHP どっち？

A. 「どちらも管理画面の小改修は対応できます。案件の技術に合わせます。最初の60日は通せる案件優先です。」

### Q. 設計どこまでできる？

A. 「要件の整理、画面遷移、データの持ち方、テスト観点まで自分で書けます。学習用に要件定義〜テスト仕様まで通した文書もあります。実案件では顧客の言葉で範囲と受け入れ条件を先に固定します。」

### Q. ナイトワークの仕事は？

A. 「本業の現場理解があります。日払い・シフト・来店管理など、業界特有の言葉で要件を落とせるのが強みです。必要ならその領域の小規模システム改修も相談できます。」  
（※出しすぎ注意。相手が求めているときだけ。）

## 5. 次につなげる話のフック

面談や提案の最後に、1つだけ質問を置く。

- 「今回の改修の、完了条件はどれですか？」
- 「同じ系統の追加改修は続きそうですか？」
- 「既存の設計書や画面遷移図はありますか？無ければ短く整理から入れます」

これが「安く受ける人」ではなく「設計できる人」の立ち位置になる。
