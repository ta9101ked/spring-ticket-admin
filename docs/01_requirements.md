# 要件定義書（学習用）

- 文書バージョン: 1.0
- 作成日: 2026-07-23
- 位置づけ: **仮想クライアントとの要件定義を再現した学習成果物**（実案件ではない）

## 1. 背景（顧客の言葉）

社内の依頼や不具合対応を、チャットや口頭だけで回している。  
「誰が」「何を」「今どういう状態か」が分からず、同じ依頼を二重にやってしまう。

Web やシステムに詳しくないが、**一覧で見えて、追加できて、状態が分かればよい**。

## 2. 目的

小さな社内チケット管理を用意し、依頼の見える化と状態管理を可能にする。

## 3. 利用者

| 利用者 | やりたいこと |
|---|---|
| 受付担当 | 依頼を登録する |
| 担当者 | 自分／全体のチケットを見て更新する |
| 管理者 | 状態で絞り込み、完了状況を把握する |

※ 本サンプルではログイン機能は対象外（全員同じ画面を使う前提）。

## 4. 機能要件

| ID | 要件 | 優先度 |
|---|---|---|
| R-01 | チケットを一覧表示できる | Must |
| R-02 | チケットを新規登録できる | Must |
| R-03 | チケットを編集できる | Must |
| R-04 | チケットを削除できる（確認あり） | Must |
| R-05 | タイトルのキーワードで検索できる | Must |
| R-06 | ステータスで絞り込みできる | Must |
| R-07 | キーワードとステータスを組み合わせて検索できる | Should |
| R-08 | チケット番号は重複できない | Must |
| R-09 | 必須項目が空のまま登録できない | Must |
| R-10 | 存在しないチケットを開こうとしたら分かりやすく知らせる | Should |

## 5. データ項目（顧客向け表現）

| 項目 | 説明 | 必須 |
|---|---|---|
| チケット番号 | 人が見て分かる管理番号 | 必須・重複不可 |
| 件名 | 何の依頼か | 必須 |
| 状態 | 未着手 / 対応中 / 完了 | 必須（選択） |
| 優先度 | 低 / 中 / 高 | 必須（選択） |
| 依頼者名 | 誰からの依頼か | 必須 |

## 6. 状態の定義

| 画面表示 | システム値 | 意味 |
|---|---|---|
| Open | `Open` | 未着手 |
| In Progress | `In_Progress` | 対応中 |
| Done | `Done` | 完了 |

## 7. 非機能（学習範囲）

- ローカル PC で動作すればよい
- 同時利用・本番セキュリティは対象外
- 1画面あたりの操作は直感的であること（専門用語を減らす）

## 8. 対象外

- 会員ログイン
- メール通知
- スマホ専用アプリ
- 外部システム連携

## 9. 受け入れ条件（顧客視点）

1. 一覧で登録済みチケットが見える
2. 新規登録後、一覧に増える
3. 編集後、同じ番号の内容が変わる（行が増えない）
4. 削除確認後、一覧から消える
5. 状態「対応中」だけに絞れる
6. 空欄登録でエラーが出る
7. 同じチケット番号で二重登録できない
